# Activity Summary

## Overview

这个 skill 用来稳定完成“真实线下技术活动检索与情报整理”。

它的目标不是把所有活动都抓进来，而是：

1. 先找到真实活动。
2. 再筛掉营销、伪科技、空泛活动。
3. 最后用稳定、好扫读的模板输出。

这个 skill 特别适合下面这类任务：

- 每日或每周检索粤港澳硬核技术活动
- 想优先看深圳，但接受香港等周边城市的高含金量活动
- 既要“宁缺毋滥”，又不想因为格式要求太死而总是 0 条
- 希望英文活动自动带中文解释
- 希望深圳空窗时补一个简短候补链接区

## When to Use

出现以下任一情形时优先使用：

1. 用户要检索活动、讲座、meetup、研讨会、开放日、学术报告。
2. 用户明确强调“不要编造”“必须有官方链接”“只要真实活动”。
3. 用户要按时间窗检索，比如“本周”“两周内”“每日推送”“每周汇总”。
4. 用户要聚焦前端、后端、HCI、数据可视化、AI、大模型、Agent、机器人、具身智能、计算机科研。
5. 用户要输出成可复用模板，而不是一次性的聊天回答。

## Do Not Use

下面这些场景不要强行套用这个 skill：

1. 用户只是要一条临时活动推荐，不需要系统筛选。
2. 用户只要旅游、演出、展览，不关心技术硬度。
3. 用户只是问某个学校或会议的单个事实。

## Core Principle

这个 skill 的核心原则是：

1. **真实性优先于完整性**
2. **先宽搜候选，再硬筛主清单**
3. **主清单和候补区必须分开**
4. **不确定就明说，不猜**

“更容易搜到真活动”不等于放松真实标准，而是：

- 不再要求每条活动先补齐全部辅助字段才能收录
- 不再为了证明“搜过很多平台”而牺牲主要输出
- 允许把“接近但没过线”的活动放进候补链接，而不是硬塞进主清单

## Scope

默认地域范围：

- 深圳
- 广州
- 珠海
- 香港
- 澳门

默认时间范围：

- 从本周一 00:00 到下周日 23:59

默认固定出发点：

- 深大地铁站 A3/A4 出口

## Hard Rules

### 正式主清单收录底线

只有同时满足以下条件，活动才允许进入正式主清单：

1. 线下真实活动
2. 公开网页可核验
3. 有明确活动名称
4. 有明确时间
5. 有明确城市或地点
6. 有明确主题说明
7. 与计算机、AI、开发者、科研有关
8. 面向公众、学生、开发者或科研人群可见

### 优先收录类型

1. 高校计算机或 AI 学院学术讲座、Workshop、学者报告、可旁听研讨会
2. CCF、ACM、IEEE、顶级开源社区、技术委员会主办的纯技术活动
3. Luma、Meetup、GitHub、开源社区上的纯技术开发者 meetup
4. 香港八大和岭南大学的研究硕博、RA、开放日、教授面对面咨询
5. 计算机发展史、AI 算力、数字技术、机器人、具身智能相关专业展

### 永久排除类型

1. 厂商产品宣讲
2. 商业峰会、招商论坛、老板圆桌、政企高管论坛
3. 文艺、市集、徒步、亲子科普
4. 培训招生、训练营、集训营、孵化器、圈层收费局
5. 纯概念空谈、跨行业拼盘、无明确技术主题活动
6. 无公开时间地点的活动
7. 纯私密闭门、不可公开参加的活动
8. 非计算机相关数理化研讨会，除非与 AI、机器人、HCI、计算材料、计算成像等直接相关

## Hardness Rubric

优先判定为“硬核”的信号：

1. 有代码、Demo、命令行、系统实现、算法推导、实验方法、模型训练或部署
2. 主题明显聚焦前端、Vue3、后端、全栈、软件工程、HCI、人机交互、数据可视化、AI、大模型、Agent、机器人、具身智能、AGI、计算机科研
3. 主办方或讲者明显具备学术或技术权威性

如果活动只是“科技”“创新”“AI 赋能”但没有具体技术内容，默认不要进主清单。

## City Policy

1. 深圳：允许高含金量和中含金量
2. 广州、珠海、香港、澳门：只保留高含金量

如果非深圳城市的活动只是“还行”，不要为了凑数收录。

## Search Workflow

按这个顺序工作：

1. 先确定时间窗和城市范围
2. 从高校、学术组织、Luma/Meetup、官方技术社区开始抓第一批候选
3. 对每个候选核验：时间、地点、主题、公开性
4. 再判断是否足够硬核
5. 通过者进入主清单
6. 接近但未过线者进入候补池
7. 最后再补交通、报名、旁听、费用等辅助字段

重点：**不要倒过来做**。  
不要先为了补交通、公众号、出口、票价而把真正的活动漏掉。

## Source Priority

优先把下列来源当作候选池：

- Luma、Meetup
- InfoQ、极客时间、SegmentFault、掘金、OSC、CSDN 线下活动
- 腾讯、阿里、字节、华为技术社区
- AWS 社区、OceanBase、Deepin、CANN
- LangChain、LLM 社区、PyTorch、TensorFlow、向量数据库、机器人社区
- CNCF、K8s、Go、Rust、WebAssembly
- CCF、ACM、IEEE
- 南科大、深大、港中深、华工、暨大、澳大官网和公众号
- 香港八大与岭南大学招生官网
- 前端早早聊、freeCodeCamp、OpenCloudOS、GitCode、开源之夏
- 大湾区 AI、前端、Agent、具身智能社区

使用时优先读：

- [references/source-checklist.md](references/source-checklist.md)

## Candidate Handling

活动只允许落在三个桶里：

1. **正式主清单**
   - 满足真实、公开、硬核条件
2. **候补链接**
   - 真实，但主题不够纯、硬度略弱、或偏产业/产品
3. **直接剔除**
   - 商业、虚、私密、无时间地点、非技术

不能把候补混进主清单，也不要把主清单压成 0 条后完全不告诉用户附近还有什么接近项。

更具体的判断规则见：

- [references/filtering-rubric.md](references/filtering-rubric.md)

## Output Contract

输出必须使用固定五层结构：

1. `# 检索结果`
2. `# 快速卡片`
3. `# 完整档案`
4. `# 候补链接`
5. `# 备注`

详细模板见：

- [references/output-template.md](references/output-template.md)

### 特别规则

1. 英文活动名后必须补一行“中文解释”
2. 快速卡片和完整档案的活动顺序必须一致
3. 深圳主清单为 0 时，必须优先给 `深圳候补（可自行判断）`
4. 候补区每条只保留短格式，不展开长解释

## Field Rules

1. 日期统一 `YYYY-MM-DD HH:MM`
2. 总时长不明确，写“官方未明确”
3. 报名方式不明确，写“原文未说明”
4. 校外能否旁听不明确，写“官方未明确，建议联系主办方确认”
5. 费用不明确，不要猜
6. “我能学到什么”必须写具体，不要写空话

## Traffic Rules

默认起点是深大地铁站 A3/A4 出口。

1. 深圳本地优先给地铁或地铁+步行方案
2. 广州、珠海：只保留单程大致 2.5 小时内、费用大致不高的高含金量活动
3. 香港：只保留单程大致 1 小时内可达、费用合理的高含金量活动；特别硬核可略放宽，但必须标“估算”
4. 澳门：只保留全程大致 3 小时内可达、费用合理的高含金量活动
5. 交通细节不是一票否决项；查不到精确信息时可以写“基于公开交通信息估算”

## Quality Bar

交付前做四个检查：

1. 有没有把明显营销会混进主清单
2. 有没有漏写英文活动的中文解释
3. 有没有主清单为空却没给深圳候补
4. 有没有把不确定的信息写成确定语气

## Writing Style

输出风格必须：

1. 稳定
2. 易扫读
3. 不像散文
4. 不像营销稿
5. 不夸张
6. 不装懂

## Minimal Good Outcome

即使本轮只找到 1-3 场主清单活动，也算成功；只要：

1. 活动是真的
2. 链接能点
3. 技术主题说得清
4. 模板整齐
5. 候补区有用

这个 skill 的目的不是“每次都很多”，而是“每次都靠谱、好用、好读”。
